# 从 Pi Agent 借鉴 MygoCode 的轻量改进建议

> 目的：用于日常实习项目展示。优先补齐「安全边界、可演示性、可解释性」，不追求复刻完整产品。本文基于 2026-07-19 对 [Pi Agent Harness](https://github.com/earendil-works/pi) 的 README 与使用文档，以及本仓库现有实现的对照。

> 实施入口：[PI_改进实施规格.md](./PI_改进实施规格.md)。当前状态为“仅完成规格与测试设计，尚未实现功能”；下面的 P0/P1/P2 建议必须按规格中的验收标准逐项落地。

## 先说结论

MygoCode 的底层能力已经比 Pi 更丰富：已有多模型统一接口、MCP 延迟加载、Plan Mode、权限规则、Hooks、Skills、记忆、上下文压缩、Worktree 和多 Agent。后续不应为了“功能多”继续堆叠。

值得借鉴的是 Pi 将复杂能力收敛为几个用户容易理解、容易演示的工作流。建议只做下面三项，按优先级推进。

## 建议一：项目本地配置增加“信任门槛”

**优先级：P0；预计 0.5～1 天。**

### 现状与风险

项目启动时会加载 `<project>/.mygocode/config.yaml`、`config.local.yaml`、项目级 commands 和 skills。与此同时，Hooks 支持执行命令和 HTTP 请求（`internal/hooks/hooks.go`）。这意味着用户首次进入一个陌生仓库时，项目配置已经具备影响本机的能力。

现有权限系统主要保护 Agent 发起的工具调用；它不能替代“是否信任这个项目自带配置”的启动前判断。

### 借鉴 Pi

Pi 在加载项目本地 extensions、skills、prompts 等资源前要求用户确认信任，并将选择保存为项目级决定。

### 最小实现

1. 首次发现项目级 `.mygocode/` 配置时，在 TUI 显示一次确认：`信任此项目的本地配置吗？`
2. 未信任时，仅加载全局配置；跳过项目 Hooks 和项目 commands，或将它们标为禁用。
3. 将决定存入用户目录，例如 `~/.mygocode/trusted-projects.json`；提供 `/trust` 查看或修改。
4. 添加两组测试：未信任时不执行项目 Hook；信任后项目配置正常生效。

### 面试可讲的点

“模型工具权限和项目供应链信任是两层不同问题：前者约束 Agent 行为，后者避免陌生仓库在启动阶段注入工作流。”

## 建议二：把会话从“可恢复”升级为“可分支”

**优先级：P1；预计 1～2 天。**

### 现状

当前会话已用 JSONL 持久化，支持列表和恢复（`internal/session/`、`/resume`），并在压缩时保存边界。这是很好的基础；但当用户想对同一个需求尝试两种方案时，只能另开会话或手动复制上下文。

### 借鉴 Pi

Pi 提供 `/fork`、`/clone` 和 `/tree`：用户可从旧消息重新开始，保留原方案，不会污染当前上下文。

### 最小实现

只实现 `/fork`，不要一开始做完整树形 UI：

1. `/fork` 复制当前会话 JSONL 到新会话 ID，并在首条 system 记录中写入 `parent_session_id` 和 `forked_at`。
2. `/session info` 显示当前会话 ID、父会话 ID、消息数与文件大小。
3. 在 `/resume` 列表中用简短标记显示“从哪个会话分出”。
4. 为复制完整性、父子元数据和恢复分支会话增加单元测试。

这项功能代码量小，但演示效果很好：可以展示 Agent 在“方案 A / 方案 B”之间保持独立上下文的能力。

## 建议三：增加会话导出，沉淀可交付证据

**优先级：P2；预计 0.5～1 天。**

### 借鉴 Pi

Pi 支持将会话导出为 HTML 或 JSONL，便于复盘、分享与问题复现。

### 最小实现

当前 JSONL 本身已存在，因此只需补一个只读导出层：

1. 增加 `/export [path]`，将当前会话转换为单文件 HTML；默认输出到 `.mygocode/exports/`。
2. HTML 仅包含会话元数据、用户/助手消息、工具名和截断后的工具结果；对 HTML 文本做转义，不嵌入密钥或配置原文。
3. 同时保留原始 JSONL 路径，作为调试与恢复格式。

它适合在简历中展示“我如何定位问题、调用工具、验证结果”，也方便面试前准备一份真实运行记录。

## 不建议现在做

| Pi 的能力 | 不做的原因 |
| --- | --- |
| 完整 TypeScript Extension API、npm/git 扩展安装 | MygoCode 已有 Skills、Markdown commands、Hooks 和 MCP；再造一套插件生态会扩大维护面。 |
| `/tree` 可视化会话树 | 在仅有 `/fork` 前没有足够收益，先保证分支会话可靠。 |
| 支持更多模型厂商 | 当前 Anthropic、OpenAI 与 OpenAI-compatible 已覆盖演示所需，优先打磨稳定性。 |
| Docker/微虚拟机容器化 | 现有权限与 Sandbox 已是项目亮点；跨平台容器方案投入大且不适合短期实习项目。 |
| 成本精确核算与在线模型目录 | 需要持续维护价格和模型元数据，容易产生不准确的展示数据。 |

## 推荐的一周收尾节奏

具体的 spec、验收标准和测试编号见 [PI_改进实施规格.md](./PI_改进实施规格.md)，执行顺序保持 P0 → P1 → P2：

1. **P0（0.5～1 天）**：实现项目级配置“信任门槛”，先完成未信任/信任/撤销测试。
2. **P1（1～2 天）**：实现 `/fork` 与会话父子元数据，完成复制完整性和恢复分支测试。
3. **P2（0.5～1 天）**：实现 `/export`，完成 HTML 转义、脱敏、截断和只读性测试。
4. **收尾**：运行 `go test ./...`，再更新 README 和简历表述；未通过测试的功能只写“计划中”。
